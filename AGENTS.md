# Agent 工作规约

## 语言规则

- 始终使用简体中文回复用户。
- 代码注释、文档说明、交互说明和实现说明默认使用简体中文。

## Skill 使用流程

### 1. 需求与产品设计前

必须先使用 `brainstorming` skill 进行需求澄清，不得跳过。

执行要求：

- 一次只问一个关键问题。
- 优先使用选择题形式。
- 提出 2-3 种方案，并明确推荐方案与理由。
- 在进入设计文档或代码实现前，必须获得用户确认。

适用场景：

- 新建功能模块。
- 修改现有产品流程。
- 新增或调整页面、组件、交互、状态、数据流。
- 从 0 到 1 撰写 PRD 或重写 PRD。

### 2. UI/UX 设计与审查

涉及页面、组件、布局、视觉风格、交互细节、响应式、可访问性或视觉优化时，使用 `impeccable` skill。

执行要求：

- 先分析目标用户、使用场景和现有设计系统。
- 优先沿用已有组件、设计 token、布局规范和交互习惯。
- 输出方案时说明页面结构、核心交互、状态变化、异常情况。
- 不得只做装饰性页面；应优先完成真实可用的业务界面。

### 3. PRD 新建或修订

新建 PRD、重写 PRD、统一 PRD 结构或整理 PRD 时，使用 `write-a-prd` skill。

新建 PRD 或从 0 到 1 重写 PRD 时：

- 必须先完成 brainstorming。
- 必须先输出草案给用户审查。
- 必须明确说明：`这是草案，请审查后再写入文件`。
- 用户明确说“可以写入”“没问题”或同等确认后，才允许写入文件。

整理既有 PRD、统一格式、批量修订存量文档时：

- 如用户已明确授权继续执行，可直接在现有文件上修改。
- 不需要逐篇再次走草案审查。

### 4. 代码实现

当方案、PRD 或设计稿已确认，用户要求开始实现功能、修改前端组件、落地 Figma 设计或按规范写代码时，使用 `implement-with-design-tokens` skill。

执行要求：

- 编码前读取项目规范和 `theme-config.css`。
- 直接使用已有颜色、字体和设计 token 变量。
- 不得硬编码颜色、字体、间距、圆角等已有 token 能覆盖的样式。
- 不得自行封装中间 token 变量。
- 不得新增组件级暗色模式逻辑。
- 主要代码块需要添加简洁中文功能标注。
- 优先复用现有组件、页面结构、工具函数和数据流模式。

## 文档落盘规则

### 1. 模块目录

新增或整理模块文档与代码时，必须在对应产品目录下创建功能模块文件夹。

模块文件夹内统一放置：

- `PRD.md`
- `交互文档.md`
- `功能实现文档.md`
- 该模块相关代码文件

示例：

```text
PRD/Student PC/在线学习/PRD.md
PRD/Student PC/在线学习/交互文档.md
PRD/Student PC/在线学习/功能实现文档.md
PRD/Student PC/在线学习/<代码文件>
```

不得继续引用已废弃旧路径或旧模板。

### 2. PRD 文档格式

PRD 必须严格按照仓库既有 `PRD/PRD模版.md` 的结构撰写，不得自行更换章节骨架或改用其他模板。

PRD 标题格式：

```text
<产品端><功能模块> PRD Vx.y
```

必须包含以下章节：

- 前言
- 需求背景
- 流程图
- 原型图与规则定义
- 异常流与边界条件
- 数据埋点
- 功能清单与排期
- 待办事项

可选章节仅允许：

- 性能要求
- 附录

格式要求：

- 流程图必须使用 mermaid 格式。
- 原型图必须使用 ASCII 艺术展示界面布局。
- 每次修改 PRD 后必须更新版本变更日志。
- 版本号格式为 `V1.0`、`V1.1`、`V2.0`。
- 版本变更日志必须记录变更日期、变更人、变更内容。

### 3. 交互文档格式

`交互文档.md` 面向产品、设计和研发，说明“用户如何使用页面”。

必须详细说明：

- 页面入口。
- 页面展示规则。
- 按钮点击与跳转。
- 弹窗打开、关闭、确认、取消逻辑。
- 表单、作答或操作交互。
- 结果页、详情页或返回逻辑。
- 加载、空状态、错误状态、禁用状态。

不得只写概要流程。

### 4. 功能实现文档格式

`功能实现文档.md` 面向研发，说明“实现了哪些状态和数据流”。

仅保留以下三块：

- 实现范围
- 核心状态
- 数据流

不得写入以下内容：

- 页面跳转或流转机制。
- 改动文件清单。
- 设计 token 规则。
- 验证过程。
- 样式检查。
- 浏览器预览结果。

## 实现完成后的交付要求

代码实现完成后，必须同步补齐或更新：

- `PRD.md`，如本次需求涉及 PRD 变更。
- `交互文档.md`，说明用户如何使用页面和各类交互规则。
- `功能实现文档.md`，说明实现范围、核心状态和数据流。

如果只修改技术实现且不改变产品规则，应至少更新 `功能实现文档.md`；如影响用户路径或页面交互，必须同时更新 `交互文档.md`。

## 禁止行为

- 未经用户审查直接写入或覆盖新建 PRD。
- 跳过 `brainstorming` 直接开始设计或实现。
- 在未确认需求细节时做高风险假设。
- 将同一模块的 PRD、交互文档、功能实现文档和相关代码分散到不同目录。
- 未按仓库既有 PRD 模版撰写文档。
- 继续引用已废弃旧路径或旧模板。
- 实现前绕过设计 token 规范。
