# 按设计 Token 实现代码

用于“方案已经确定，开始编码实现”的阶段。目标是让实现结果符合项目设计规范，而不是重新发散设计方案。

## 必读文件

开始编码前必须读取：

1. `AGENTS.md`：确认项目级语言、PRD、协作和禁止事项。
2. `theme-config.css`：读取文件顶部“设计 token 使用规范”和现有 CSS 变量。
3. 与任务相关的现有组件、样式、页面文件：优先复用已有模式。

## 工作流程

1. 判断任务是否已经进入实现阶段。
   - 如果用户仍在讨论方案、PRD、需求边界或设计取舍，先不要编码。
   - 如果用户明确说“开始实现”“按方案写代码”“落地这个页面/组件/功能”，使用本 Skill。

2. 先定位现有实现。
   - 查找相近组件、页面、样式文件和 token 使用方式。
   - 优先复用项目已有组件、工具函数、样式组织方式。

3. 编码时遵守 token。
   - 颜色统一直接使用 `theme-config.css` 中已有 CSS 变量或其 Tailwind 映射。
   - 不在业务组件里直接写 Hex、RGB、HSL 色值。
   - 不自行封装 `--brand`、`--text`、`--panel`、`--bg` 等中间变量。
   - 不新增 `success` / `error` / `warning` 三组语义色变量。
   - 成功态用 `green`，错误态用 `red`，警告态用 `orangeRed`。
   - 不编写组件级暗色模式颜色逻辑；页面明暗模式由系统统一控制。

4. 字体按规范实现。
   - 默认字体族使用 `PingFang SC`。
   - 常用字号/行高遵循：`12/18`、`12/20`、`14/22`、`16/24`、`20/28`、`24/32`、`30/38`、`38/46`。
   - 字重使用 `400`、`500`、`600`、`700`。
   - 页面标题、区块标题、正文、表单、按钮、标签的字号选择按 `theme-config.css` 顶部规范执行。

5. 状态样式按规范实现。
   - 主品牌默认态优先用 `--brand-wzj6`。
   - Hover / Active 优先用 `--brand-wzj7`。
   - 品牌弱背景用 `--brand-wzj1` / `--brand-wzj2`。
   - 普通文本用 `--font-wzj1`，次级文本用 `--font-wzj3`，辅助文本用 `--font-wzj4`。
   - 禁用文本用 `--font-disable`，选中文本用 `--font-selected`。
   - 文字和图标颜色优先使用 `font` 色系。
   - 背景、卡片、分割线、边框和控件底色优先使用 `gray` 色系。
   - 边框/分割线用 `--gray-wzj3`。
   - 控件默认灰底用 `--gray-wzj4`，控件 Hover 灰底用 `--gray-wzj5`。
   - 部分列表项和卡片 Hover 背景可以用 `--gray-wzj3`。

6. 为主要代码块添加功能标注。
   - 在组件、页面区域、样式分组、核心函数、状态处理、数据流处理前添加简短中文注释。
   - 注释说明“这一块做什么功能”，不要解释显而易见的语法。
   - 不逐行注释，不写空泛注释。
   - 如果文件已有分区注释，沿用原有风格。

## 代码格式约定

- 使用项目现有技术栈和样式写法，不为了使用 token 引入新框架。
- 如果是 Tailwind 写法，优先使用项目已有 `--color-*` 映射；没有合适映射时使用任意值语法，例如 `text-[var(--font-wzj1)]`。
- 如果是 CSS / CSS Module，直接使用 `var(--token-name)`。
- 不为了“语义化”再定义一层局部颜色变量；可读性通过类名和组件结构表达。
- 组件状态要显式覆盖 `hover`、`active/selected`、`disabled`、`loading` 等任务涉及的状态。
- 文案、注释和说明使用简体中文。

## 实现完成后的文档

代码实现完成并验证后，除非用户明确说不需要落盘，必须将代码文件、新增或更新的两份文档，和对应模块 PRD 放在同一个模块文件夹中。

推荐目录结构：

1. PRD：`PRD/<产品端>/<功能模块>/PRD.md`
2. 交互文档：`PRD/<产品端>/<功能模块>/交互文档.md`
3. 功能实现文档：`PRD/<产品端>/<功能模块>/功能实现文档.md`
4. 代码文件：`PRD/<产品端>/<功能模块>/<代码文件>`

如果当前模块 PRD 仍是旧结构，例如 `PRD/Student PC/在线学习.md`，应先迁移为 `PRD/Student PC/在线学习/PRD.md`，再把交互文档、功能实现文档和相关代码放入同一文件夹。

交互文档需要面向产品、设计和研发说明“用户如何使用页面”，按 PRD 内容详细描述：

- 页面入口：说明从哪些入口进入哪些页面。
- 页面规则：说明页面展示规则、业务规则、状态规则、滚动规则、空状态和禁用规则。
- 按钮跳转：逐项说明点击某个按钮、Tab、列表项或节点后进入哪个页面。
- 弹窗交互：说明点击哪个入口打开弹窗，点击关闭、遮罩或快捷键后如何关闭。
- 作答交互：如涉及练习或题目，说明选项选择、提交、下一题、查看结果、提示、解析、答案等按钮规则。
- 结果交互：说明结果页不同按钮分别返回哪里，是否重新开始当前任务。

功能实现文档需要面向研发说明“实现了哪些状态和数据流”，只保留以下三块：

- 实现范围：说明本次原型或功能覆盖哪些业务链路。
- 核心状态：列出页面状态、业务状态、组件状态、数据集合及其用途。
- 数据流：说明数据如何初始化、渲染、被用户操作更新，以及如何进入结果或回写状态。

功能实现文档不要写以下内容：

- 不写按钮跳转和页面流转机制，这些统一放到交互文档。
- 不写改动文件清单。
- 不写设计 token 使用规则。
- 不写验证方式、样式检查、浏览器预览结果等过程性内容。

## 自检

提交最终结果前检查：

1. 新增或修改的 UI 代码没有直接写死颜色值，除非是非主题资产且有明确理由。
2. 没有新增局部颜色封装变量，也没有新增 `success` / `error` / `warning` token。
3. 新增状态样式覆盖了任务要求的 Hover、Selected、Disabled 等状态。
4. 没有编写组件级暗色模式颜色逻辑。
5. 主要代码块已添加中文功能标注。
6. 已在对应模块 PRD 文件夹中输出代码文件、交互文档和功能实现文档，或用户明确不需要文档。
7. 如改动前端界面，按项目条件运行类型检查、测试或本地预览验证。
